# Demand Elegance (Balanced)

For non-trivial changes: pause and ask "is there a more elegant way?"

## When to Use

- Non-trivial feature implementations
- When a fix feels hacky or forced
- Before presenting your work to the user
- **Skip this** for simple, obvious fixes — don't over-engineer

## Procedure

1. **Pause before committing** — After implementing, step back and review
2. **Ask the elegance question** — "Is there a more elegant way to do this?"
3. **Apply the hindsight test** — "Knowing everything I know now, implement the elegant solution"
4. **Challenge your own work** — Before presenting it to the user
5. **Balance** — If the simpler approach is genuinely better, keep it. Elegance ≠ complexity

## Rules

- Don't over-engineer simple fixes in the name of elegance
- Elegance means clarity, maintainability, and minimal complexity
- If a refactor doesn't meaningfully improve the code, skip it
- The goal is code a staff engineer would be proud of, not clever code
